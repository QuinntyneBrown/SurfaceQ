# L2 - Detailed Requirements

## L2-001: `generate` subcommand writes public-api.ts
**Traces to:** L1-002

The `generate` subcommand shall run discovery and write (overwriting) the generated content to the resolved `entryFile` path.

**Acceptance Criteria:**
1. Given a valid Angular library with an `ng-package.json`, when the user runs `surfaceq generate`, then the tool must write the generated content to the path specified by `entryFile` and exit with code `0`.
2. Given an existing `public-api.ts`, when `generate` runs, then the file must be overwritten with the newly rendered content.
3. Given `--project <path>` pointing to a project directory, when `generate` runs, then the tool must resolve `ng-package.json` within that directory and operate against it.
4. Given `--project <path>` pointing directly to an `ng-package.json` file, when `generate` runs, then that manifest must be used.

---

## L2-002: `check` subcommand verifies drift
**Traces to:** L1-002, L1-006

The `check` subcommand shall compare the would-be generated output against the on-disk `public-api.ts` without writing, and exit non-zero on drift.

**Acceptance Criteria:**
1. Given the on-disk `public-api.ts` matches the generated output byte-for-byte, when `surfaceq check` runs, then the tool must exit with code `0` and must not modify the file.
2. Given the on-disk `public-api.ts` differs from the generated output, when `check` runs, then the tool must exit with code `1` and print a concise drift message to stdout without printing the full diff.
3. Given an error occurs during execution (missing manifest, parse failure, etc.), when `check` runs, then the tool must exit with code `2`.

---

## L2-003: `diff` subcommand prints unified diff
**Traces to:** L1-002

The `diff` subcommand shall emit a human-readable unified diff between the existing and would-be generated output without writing to disk.

**Acceptance Criteria:**
1. Given files match, when `surfaceq diff` runs, then the tool must exit with code `0` and print no diff.
2. Given files differ, when `diff` runs, then the tool must print a unified diff to stdout and exit with code `1`.
3. Given an execution error, when `diff` runs, then the tool must exit with code `2`.
4. Given any invocation of `diff`, when the command completes, then the on-disk `public-api.ts` must be unchanged.

---

## L2-004: Project locator walks upward for ng-package.json
**Traces to:** L1-003

The project locator shall search for `ng-package.json` starting at the working directory (or `--project` path) and walking toward the filesystem root.

**Acceptance Criteria:**
1. Given `ng-package.json` exists in the starting directory, when the locator runs, then it must resolve that file.
2. Given `ng-package.json` exists in an ancestor directory, when the locator runs from a nested subdirectory, then it must resolve the nearest ancestor's manifest.
3. Given no `ng-package.json` exists in the path to the filesystem root, when the locator runs, then the tool must exit with code `2` and log an error message identifying the search path.

---

## L2-005: entryFile resolution and fallback
**Traces to:** L1-003, L1-011

The `entryFile` field of `ng-package.json` (resolved relative to the manifest's directory) determines both the output path and the scan root.

**Acceptance Criteria:**
1. Given `ng-package.json` specifies `entryFile: "src/public-api.ts"`, when the tool resolves paths, then the output path must equal `<manifestDir>/src/public-api.ts` and the scan root must equal `<manifestDir>/src`.
2. Given `ng-package.json` does not contain an `entryFile` field, when the tool resolves paths, then it must fall back to `src/public-api.ts` and log an informational message.
3. Given `ng-package.json` is malformed JSON, when the tool runs, then it must exit with code `2` and log an error identifying the file and the parse error.
4. Given the resolved `entryFile` directory does not exist, when the tool runs, then it must exit with code `2` with a clear error message.

---

## L2-006: Source file walker exclusions
**Traces to:** L1-003

The source file walker shall recursively enumerate `.ts` files under the scan root, excluding `*.spec.ts`, `*.stories.ts`, any file literally named `index.ts`, the `entryFile` itself, and any file under a `node_modules` directory.

**Acceptance Criteria:**
1. Given a scan root containing a mix of `.ts`, `.spec.ts`, `.stories.ts`, and `index.ts` files, when the walker runs, then the returned list must include only non-excluded `.ts` files.
2. Given a scan root containing a `node_modules` subdirectory with `.ts` files, when the walker runs, then no file under `node_modules` must be returned.
3. Given a scan root that contains the `entryFile`, when the walker runs, then the `entryFile` itself must not be returned.
4. Given the scan root contains zero `.ts` files after exclusions, when `generate` runs, then a header-only `public-api.ts` must be written, an informational message logged, and exit code `0` returned.

---

## L2-007: Deterministic file ordering
**Traces to:** L1-005

The walker shall sort discovered files by relative path in a deterministic, cross-platform order.

**Acceptance Criteria:**
1. Given the same set of input files, when the walker runs on Windows, Linux, and macOS, then the returned ordering must be identical.
2. Given the tool runs twice against unchanged source, when outputs are compared, then they must be byte-identical.

---

## L2-008: Symbol discovery categories
**Traces to:** L1-004

The discovery engine shall identify named exports across documented categories: Angular decorators (Component, Directive, Pipe, Injectable, NgModule), standalone primitives and functional guards/interceptors/resolvers, type aliases, interfaces, enums and const enums, exported const/let/var including `InjectionToken` instances, exported functions and classes, file-level `export { X } from '...'` re-exports, and type-only exports.

**Acceptance Criteria:**
1. Given a file contains `export class FooComponent` decorated with `@Component`, when discovery runs, then `FooComponent` must appear as a value export.
2. Given a file contains `export interface Bar`, when discovery runs, then `Bar` must appear as a type-only export.
3. Given a file contains `export type Baz = ...`, when discovery runs, then `Baz` must appear as a type-only export.
4. Given a file contains `export const TOKEN = new InjectionToken<...>()`, when discovery runs, then `TOKEN` must appear as a value export.
5. Given a file contains `export { X } from './sub'`, when discovery runs, then `X` must be emitted in the generated output with the correct re-exported form.
6. Given a file contains `export type { Foo }`, when discovery runs, then `Foo` must be emitted as a type-only export.

---

## L2-009: Wildcard re-export expansion
**Traces to:** L1-001, L1-004

Files containing `export *` wildcard re-exports shall be resolved through the TypeScript Compiler API into the underlying named symbols and emitted explicitly in the generated output.

**Acceptance Criteria:**
1. Given a file contains `export * from './inner'` where `./inner` exports `A` and `B`, when the tool runs, then the generated output must emit `A` and `B` as named exports (via the file that originally contained them) with no wildcard form.
2. Given output is inspected, when searched for `export *`, then no occurrence must be present.

---

## L2-010: Default export skip with warning
**Traces to:** L1-011

Default exports shall be skipped and a warning logged to stderr; the run must continue and exit `0`.

**Acceptance Criteria:**
1. Given a file contains `export default ...`, when the tool runs, then no default export must appear in the generated output.
2. Given a default export is skipped, when the tool runs, then a warning prefixed `warn:` identifying the offending file (as a relative path) must be written to stderr.
3. Given only default exports exist in a file, when the tool runs, then the overall exit code must still be `0`.

---

## L2-011: Output format header and structure
**Traces to:** L1-005

The generated file shall begin with the documented header comment, group symbols from the same source file into one `export { ... } from '...'` statement, emit files in discovered order, preserve source-order within a file, use relative POSIX-style module specifiers without a `.ts` extension, and end with a trailing newline. Indentation shall use two spaces.

**Acceptance Criteria:**
1. Given any generation run, when the output is inspected, then the first lines must contain the SurfaceQ-generated header comment block instructing readers not to edit by hand.
2. Given a file exports `A`, `B`, and `C` as values, when the tool renders, then a single `export { A, B, C } from '<relative>'` statement must be emitted.
3. Given files are processed on Windows, when module specifiers are inspected, then they must use forward slashes and must not contain backslashes.
4. Given any emitted module specifier, when inspected, then it must not contain a `.ts` suffix and must be a relative path from the entry file's directory.
5. Given any generated file, when inspected byte-for-byte, then it must end with a single trailing newline.
6. Given a file exports both a value `X` and a type `Y`, when rendered, then two adjacent statements `export { X } from '...'` and `export type { Y } from '...'` must be emitted.
7. Given indentation is inspected inside multi-line statements, then two-space indentation must be used.

---

## L2-012: Determinism across runs and platforms
**Traces to:** L1-005, L1-006

Output must be byte-identical across runs, hosts, and operating systems for unchanged input.

**Acceptance Criteria:**
1. Given the same fixture library, when the tool runs sequentially on Windows, Linux, and macOS CI agents, then the resulting `public-api.ts` files must hash identically (SHA-256).
2. Given any two successive runs against unchanged source on the same host, when outputs are diffed, then they must be byte-identical.

---

## L2-013: Node.js sidecar lifecycle and communication
**Traces to:** L1-004, L1-010

The .NET host shall spawn a bundled Node.js sidecar process per CLI invocation, communicate via newline-delimited JSON on stdio, and pass only file paths (never file content) as parameters.

**Acceptance Criteria:**
1. Given a CLI invocation, when the tool starts, then exactly one Node sidecar process must be spawned with explicit arguments (no shell invocation).
2. Given the sidecar is launched, when its parameters are inspected, then the arguments and JSON payload must contain only file paths and a `projectRoot`, never file contents.
3. Given the CLI invocation completes, when processes are checked, then the sidecar process must be terminated.
4. Given the CLI sends a `discover` request, when the sidecar responds, then the response must include the `id` matching the request and a `result.files` array with per-file `exports` and `warnings`.

---

## L2-014: Bundled Node runtime and TypeScript package
**Traces to:** L1-007, L1-010

The NuGet package shall bundle Node.js binaries per runtime identifier (RID) and the `typescript` npm package pre-installed, requiring no `npm install` at runtime and no user-installed Node.js.

**Acceptance Criteria:**
1. Given a clean machine with no Node.js installed, when the user installs SurfaceQ via `dotnet tool install --global SurfaceQ` and runs `surfaceq generate`, then the command must succeed.
2. Given the installed package contents are inspected, when Node binaries are listed, then binaries must be present for Windows x64/ARM64, Linux x64/ARM64, and macOS x64/ARM64.
3. Given the installed package contents are inspected, when sidecar `node_modules` is listed, then `typescript` must be present and no `npm install` must be executed during any CLI run.
4. Given the installed package on disk, when measured, then the total footprint must be under 60 MB.

---

## L2-015: Cold and warm performance targets
**Traces to:** L1-008

The tool shall meet documented runtime performance targets on a modern developer laptop.

**Acceptance Criteria:**
1. Given a 50-file library and a cold tool invocation, when `surfaceq generate` runs, then it must complete in under 5 seconds.
2. Given a 200-file library, when `surfaceq generate` runs in a warm state, then it must complete in under 10 seconds.
3. Given a 500-file library, when the tool runs, then peak working set memory must remain under 512 MB.

---

## L2-016: Logging streams and verbosity
**Traces to:** L1-009

Logging shall be plain-text via `Microsoft.Extensions.Logging` with a console provider. A global `--verbosity <level>` flag shall accept `quiet`, `minimal`, `normal` (default), `detailed`, and `diagnostic`. Errors and warnings go to stderr; informational and verbose output go to stdout.

**Acceptance Criteria:**
1. Given any invocation, when log output is captured, then warnings and errors must be written to stderr and informational messages to stdout.
2. Given `--verbosity quiet`, when the tool runs without errors, then no output must be produced on stdout.
3. Given `--verbosity diagnostic`, when the tool runs, then detailed trace output including file discovery and sidecar communication must be produced.
4. Given no `--verbosity` flag is provided, when the tool runs, then `normal` verbosity must be used.
5. Given any log output, when inspected, then it must be plain text with no ANSI color codes and no JSON framing.

---

## L2-017: No network, no telemetry
**Traces to:** L1-010

The tool shall not perform any network I/O at runtime.

**Acceptance Criteria:**
1. Given any CLI invocation, when network traffic is monitored, then zero outbound connections must be observed from the SurfaceQ process or sidecar.
2. Given any invocation, when telemetry collection is audited, then no telemetry endpoint must be contacted and no update checks must be performed.

---

## L2-018: Syntax error handling
**Traces to:** L1-011

When the sidecar reports a syntax-error diagnostic, the tool shall exit with code `2` and report the offending file and line.

**Acceptance Criteria:**
1. Given a scanned file contains a TypeScript syntax error, when the tool runs, then it must exit with code `2` and print an error to stderr identifying the file (as a relative path) and line number.

---

## L2-019: File permission errors
**Traces to:** L1-011

When the target `public-api.ts` cannot be written due to filesystem permissions, the tool shall exit with code `2` and report a permissions error.

**Acceptance Criteria:**
1. Given `public-api.ts` exists and is read-only, when `surfaceq generate` runs, then it must exit with code `2` and print an error to stderr identifying the file and the permission issue.

---

## L2-020: Name collisions are not adjudicated
**Traces to:** L1-001, L1-011

When the same exported name originates from two different files, both shall be emitted; SurfaceQ shall not rename or omit either.

**Acceptance Criteria:**
1. Given two scanned files both export a symbol named `Foo`, when the tool runs, then both `export { Foo } from '...A'` and `export { Foo } from '...B'` statements must be present in the output.

---

## L2-021: Global flags surface
**Traces to:** L1-002

The CLI shall expose global `--verbosity`, `--version`, and `--help` / `-h` flags alongside every subcommand.

**Acceptance Criteria:**
1. Given `surfaceq --version`, when executed, then the SurfaceQ semantic version must be printed to stdout and the process must exit with code `0`.
2. Given `surfaceq --help` or `surfaceq -h`, when executed, then help text listing all subcommands and global options must be printed and exit with code `0`.
3. Given `surfaceq <subcommand> --help`, when executed, then subcommand-specific help must be printed and exit with code `0`.

---

## L2-022: No v1 configuration file
**Traces to:** L1-012

v1 shall not read any project-level configuration file; all behavior is driven by `ng-package.json` plus CLI flags.

**Acceptance Criteria:**
1. Given any file named `surfaceq.config.json`, `.surfaceqrc`, or similar is present in the project, when the tool runs, then it must be ignored and behavior must be identical to a run without those files.
